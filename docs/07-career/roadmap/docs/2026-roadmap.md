> 본 포스팅은 아무것도 모르는 상태에서 최신 기술 스택을 하나씩 깨부수며 나아가는 시니어 백엔드 개발자의 실전 기록입니다. "기한보다 내실"을 목표로 긴 호흡의 여정을 시작합니다.

## 🛠️ 2026 핵심 기술 스택 (Tech Stack)

| 카테고리 | 기술 스택 | 학습 및 적용 포인트 |
| :--- | :--- | :--- |
| **Backend** | .NET 10 | Minimal API 기반의 가볍고 강력한 엔진 구축 |
| **Data** | Dapper & EF Core | 레거시 DB(No FK)를 다루는 하이브리드 ORM 전략 |
| **Messaging** | Kafka & Redis | 비동기 이벤트 기반 아키텍처 및 분산 캐싱/락 |
| **Frontend** | Next.js 15+ | React 기반 UI 및 Static Export를 활용한 배포 최적화 |
| **Cloud/DevOps** | Azure & Bitbucket | 통합 CI/CD 파이프라인 및 하이브리드 클라우드 환경 |
| **Monitoring** | Grafana | 시스템 메트릭 시각화 및 통합 관제 대시보드 |

## 📅 2026 실전 로드맵

회사 업무 시간 내 투자 가능한 시간을 고려하여, 2주 간격으로 하나의 마일스톤을 달성하는 구조입니다.

| 스프린트 | 기간 (2026) | 테마 | 핵심 과업 (Task) | 목표 (Milestone) |
| :--- | :--- | :--- | :--- | :--- |
| **Sprint 1** | 05/11 ~ 05/24 | API 기초 | Minimal API 심화 학습 및 프로젝트 환경 완벽 세팅 | 현대적 API 아키텍처 이해 |
| **Sprint 2** | 05/25 ~ 06/07 | 조회 최적화 | Dapper 라이브러리 학습 및 레거시 DB 연결 테스트 | 고성능 데이터 조회 레이어 |
| **Sprint 3** | 06/08 ~ 06/21 | 명령 최적화 | EF Core Fluent API 공부 (No-FK 관계 설정) | 하이브리드 데이터 레이어 |
| **Sprint 4** | 06/22 ~ 07/05 | 백엔드 통합 | Sprint 1~3 통합 및 핵심 CRUD API 구축 완료 | 백엔드 엔진 완성 |
| **Sprint 5** | 07/06 ~ 07/19 | 프론트 기초 | Next.js 15 & React 기본 개념 학습 (컴포넌트/Props) | 프론트엔드 적응기 |
| **Sprint 6** | 07/20 ~ 08/02 | UI 연동 | Tailwind CSS 학습 및 백엔드 API 연동 실습 | UI 프로토타입 완성 |
| **Sprint 7** | 08/03 ~ 08/16 | 메시징 입문 | Kafka 개념 공부 및 로컬 도커 환경 구축 (No Code) | 분산 시스템 사고방식 전환 |
| **Sprint 8** | 08/17 ~ 08/30 | 메시징 실전 | .NET-Kafka 연동 실습 (메시지 Pub/Sub 기초) | 비동기 데이터 흐름 구현 |
| **Sprint 9** | 08/31 ~ 09/13 | 캐싱/심화 | Redis 캐싱 전략 학습 및 Kafka 혼용 로직 설계 | 시스템 성능 최적화 가동 |
| **Sprint 10** | 09/14 ~ 09/27 | CI/CD 통합 | Bitbucket-Azure Pipelines 연동 및 YAML 설정 사투 | 자동 배포 파이프라인 기초 |
| **Sprint 11** | 09/28 ~ 10/11 | 클라우드 배포 | Azure 인프라 설정 및 운영 환경 배포 검증 | 클라우드 네이티브 환경 |
| **Sprint 12** | 10/12 ~ 10/25 | 관제/회고 | Grafana 시각화 공부 및 최종 시스템 안정화 점검 | 모니터링 시스템 완성 |

## 📢 [Notice] 운영 및 일정 관리 안내

**1. 학습과 구축의 병행**
본 프로젝트는 모든 기술을 처음 접한다는 가정하에 설계되었습니다. 따라서 각 스프린트의 첫 1주는 **개념 학습(Study)**에, 나머지 1주는 **실무 구현(Build)**에 집중합니다.

**2. 일정 순연(Shift) 가능성**
실무 환경의 특성상 긴급 프로젝트나 유지보수 이슈가 발생할 경우, 해당 스프린트 일정은 최대 2~4주까지 유연하게 순연될 수 있습니다. 억지로 일정을 맞추기보다 정확한 기술 내재화를 우선합니다.

**3. 기록의 가치**
매 스프린트가 종료될 때마다 겪었던 시행착오와 해결 과정을 블로그에 기록하여, 비슷한 고민을 하는 개발자들에게 인사이트를 공유할 예정입니다.
