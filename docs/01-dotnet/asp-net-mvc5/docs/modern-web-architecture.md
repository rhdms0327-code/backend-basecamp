> Microsoft 공식 문서를 바탕으로 모놀리식, 전통적 N-계층, 그리고 현대적인 클린 아키텍처의 차이점과 장단점을 비교 분석합니다.

## 1. 주요 설계 방식 비교

### 모놀리식 (Monolithic)
- **구조**: 모든 기능이 하나의 프로젝트와 어셈블리에 포함됨.
- **장점**: 초기 개발 속도가 매우 빠르고 구조가 단순함.
- **단점**: 규모가 커지면 의존성이 꼬여 수정과 확장이 매우 어려움.

### 전통적 N-계층 (N-Tier)
- **구조**: UI → BLL(비즈니스 로직) → DAL(데이터 접근) 순서의 수직 계층화.
- **특징**: 상위 계층이 하위 계층에 종속되는 하향식 의존 구조.
- **한계**: 로직이 특정 DB 기술에 묶여 데이터베이스 없이는 단독 테스트가 어려움.

## 2. 클린 아키텍처 (Clean/Onion Architecture)

비즈니스 로직(Core)을 중심에 두고 UI와 인프라(DB)가 외부에서 중심을 바라보게 설계합니다. **의존성 역전 원칙(DIP)**이 핵심입니다.

- **Application Core**: 엔티티, 도메인 서비스, 인터페이스 등 순수 비즈니스 로직 포함. 외부 기술에 의존하지 않음.
- **Infrastructure**: DB(EF Core), 외부 API, 파일 시스템 등 구체적인 구현 담당.
- **UI Layer**: 컨트롤러, 뷰, API 엔드포인트 등 사용자 접점.

## 3. 클린 아키텍처의 장점

1. **테스트 용이성**: Core 계층은 인프라 없이도 단위 테스트(Unit Test)가 가능합니다.
2. **기술 독립성**: 데이터베이스나 프레임워크가 바뀌어도 핵심 비즈니스 로직은 수정할 필요가 없습니다.
3. **유연성**: UI 계층은 인프라의 구체적인 구현을 몰라도 되며, 필요 시 구현체만 교체할 수 있습니다.

> [!TIP]
> 엔티티는 단순한 DB 테이블의 복사본이 아니라 **비즈니스 규칙 그 자체**여야 합니다. 따라서 특정 라이브러리에 오염되지 않은 순수 C# 클래스(POCO)로 존재해야 합니다.
