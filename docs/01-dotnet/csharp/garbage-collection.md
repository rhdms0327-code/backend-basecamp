# [22장] 가비지 컬렉션 (Garbage Collection)

## 1. 루트 (Root)
- **루트**는 할당된 메모리의 위치를 참조하는 객체입니다.
- **가비지 컬렉터(GC)**는 루트에 연결된 객체들은 '살아있다'고 판단하고, 연결이 끊긴 객체들은 '쓰레기(Garbage)'로 간주하여 수거합니다.
- **루트의 종류**:
  - 정적 필드 (Static Fields)
  - 스택의 지역 변수 및 매개변수
  - CPU 레지스터, 핸들 테이블 등 현재 사용 중인 참조값들

## 2. Managed Heap의 메모리 관리
### 메모리 할당
- **순차적 배치**: 객체를 생성할 때마다 힙의 포인터를 이동시켜 바로 다음 빈 공간에 붙여넣습니다.
- **효율성**: 빈 자리를 찾는 복잡한 계산이 없어 할당 속도가 매우 빠르며, 객체들이 나란히 저장되어 CPU 캐시 효율(데이터 접근 효율)이 좋습니다.

### 메모리 해제 과정
1. **가비지 식별 (Mark)**: 루트를 기점으로 연결된 객체를 추적하여 생존 그래프를 작성하고, 포함되지 않은 객체를 가비지로 분류합니다.
2. **지우기 (Sweep)**: 가비지 객체들을 메모리에서 해제합니다.
3. **메모리 압축 (Compaction)**: 파편화를 방지하기 위해 살아남은 객체들을 메모리 앞쪽으로 밀착시킵니다. (복사 비용이 발생하므로 상황에 따라 진행)
4. **포인터 갱신 (Relocation)**: 객체 이동 시 루트가 새로운 주소를 가리키도록 업데이트합니다.

## 3. 세대별 관리 (Generational GC)
"최근 생성된 객체는 금방 죽고, 오래 살아남은 객체는 계속 살 확률이 높다"는 가설에 기반하여 힙을 3개의 세대로 나눕니다.

- **0세대 (Gen 0)**: 신규 할당된 객체들. 수집 속도가 매우 빠릅니다.
- **1세대 (Gen 1)**: 0세대 GC에서 살아남은 객체들.
- **2세대 (Gen 2)**: 1세대 GC에서도 살아남은 장기 객체들(정적 변수 등). **Full GC**가 발생하며, 검사량이 많아 프로그램이 일시적으로 멈추는 **Stop-The-World** 현상이 뚜렷하게 나타날 수 있습니다.

## 4. 성능 최적화 팁
### 객체 생성 남발 금지
- 너무 많은 객체 생성은 0세대 임계치에 빨리 도달하게 하여 GC 오버헤드를 증가시킵니다.
- **해결책**: 객체 풀링(Object Pooling)을 사용하여 빈번하게 쓰이는 객체를 재사용하세요.

### 큰 객체(85KB 이상) 주의
- 85KB 이상의 객체는 **대형 객체 힙(LOH)**에 할당되며, 이는 즉시 2세대로 간주됩니다.
- LOH는 메모리 압축을 하지 않아 파편화를 유발하고 Full GC를 강제합니다.
- **해결책**: `List<T>`의 Capacity를 미리 지정하거나, `StringBuilder`를 사용하여 불필요한 대형 객체 생성을 방지하세요.

### 참조 복잡도 낮추기
- 참조 관계가 거미줄처럼 복잡하면 GC가 생존 여부를 판단하는 데 더 많은 CPU를 소모합니다.
- **해결책**: 불필요한 양방향 참조를 피하고, 사용이 끝난 이벤트 핸들러는 반드시 해제(`-=`) 하세요.
