# ⚡ [22장] 가비지 컬렉션 (Garbage Collection)

> .NET의 핵심인 자동 메모리 관리 시스템(GC)의 동작 원리부터 세대별 관리 전략, 그리고 애플리케이션의 성능을 높이기 위한 최적화 팁을 상세히 정리합니다.

## 🧠 1. 루트 (Root)
- **루트**는 할당된 메모리의 위치를 참조하는 시작점입니다.
- **가비지 컬렉터(GC)**는 루트에 연결된 객체들은 '생존'으로 판단하고, 연결이 끊긴 객체들은 '가비지(Garbage)'로 간주하여 수거합니다.
- **루트의 종류**: 정적 필드, 스택의 지역 변수 및 매개변수, CPU 레지스터 등.

## 🏗️ 2. Managed Heap의 메모리 관리
- **메모리 할당**: 순차적으로 포인터를 이동하며 배치하여 할당 속도가 매우 빠르고 데이터 접근 효율이 좋습니다.
- **메모리 해제 과정**:
  1. **Mark**: 생존 객체를 식별합니다.
  2. **Sweep**: 가비지 객체를 해제합니다.
  3. **Compaction**: 파편화 방지를 위해 생존 객체를 앞쪽으로 압축합니다.
  4. **Relocation**: 객체 이동에 따른 주소값을 갱신합니다.

## ⏱️ 3. 세대별 관리 (Generational GC)
"최근 생성된 객체는 수명이 짧고, 오래 생존한 객체는 계속 유지될 확률이 높다"는 가설에 기반합니다.
- **0세대 (Gen 0)**: 신규 할당된 객체들로 수집 속도가 매우 빠릅니다.
- **1세대 (Gen 1)**: 0세대에서 살아남은 과도기적 객체들입니다.
- **2세대 (Gen 2)**: 장기 생존 객체들입니다. **Full GC** 발생 시 프로그램이 일시적으로 멈추는 **Stop-The-World** 현상이 두드러질 수 있습니다.

> [!WARNING]
> 85,000바이트 이상의 객체는 **LOH (Large Object Heap)**에 할당되며 즉시 2세대로 취급됩니다. LOH는 기본적으로 메모리 압축을 하지 않아 파편화에 취약하므로 주의가 필요합니다.

## 🚀 4. 성능 최적화 팁
- **객체 풀링**: 빈번하게 생성/소멸되는 객체는 풀(Pool)에 담아 재사용하여 GC 오버헤드를 줄입니다.
- **리스트 용량 지정**: `List<T>` 생성 시 `Capacity`를 미리 지정하면 불필요한 내부 배열 재할당과 LOH 유입을 막을 수 있습니다.
- **참조 관계 단순화**: 복잡한 참조 그래프는 GC의 Mark 단계 소요 시간을 증가시킵니다.
